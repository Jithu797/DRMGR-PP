📖 Key Definitions
Variable
A named container in memory that holds a value. Like a labelled jar. You put something in, look at it, change it.

Real World: Your phone's contact app stores "Riya → 9876543210". Riya is the variable name, the number is the value.
str — String
Any text in quotes. Instagram usernames, bios, captions, hashtags, DMs — all strings.
username = "riya_codes"
int — Integer
A whole number with no decimal. Follower count, likes, post count.
followers = 45000
float — Decimal Number
Numbers with decimal points. Engagement rate (4.7%), GPS coordinates, video duration.
engagement_rate = 4.7
bool — Boolean
Only two values: True or False. Verified badge, private account, dark mode, notifications — all booleans.
is_verified = True
print() and input()
print() displays output on screen. Like a WhatsApp notification appearing.
input() takes user data at runtime. Like Instagram's "enter your username" box.
🧠 Memory Model: Think of RAM as a long street with numbered houses. When you write followers = 45000, Python books a house, puts 45000 inside, and sticks label "followers" on the door. Whenever you type followers, Python goes to that door and retrieves the value.

✨ f-strings — Clean Text Formatting
What is an f-string? Instead of joining strings with + signs (messy, error-prone), f-strings let you embed variable names directly inside text using {} braces. Every Python developer uses these. Instagram's backend uses the same pattern.
# Old way — messy: print("Hello " + name + ", you have " + str(followers) + " followers") # f-string — clean: print(f"Hello {name}, you have {followers:,} followers") # Expressions inside {}: print(f"Verified: {'✅' if followers > 10000 else '❌'}")
# Number formatting: 45000 → f"{45000:,}" = "45,000" 0.047 → f"{0.047:.1%}" = "4.7%" 3.14159 → f"{3.14:.2f}" = "3.14" # Multi-line f-string: msg = f""" Name: {name} Followers: {followers:,} Verified: {is_verified} """

def check_password(pwd): score = 0 # Comparison operator >= if len(pwd) >= 8: score += 1 # String method + membership + loop if any(c.isupper() for c in pwd): score += 1 if any(c.isdigit() for c in pwd): score += 1 if any(c in "@#$!%" for c in pwd): score += 1 # if / elif / else if score <= 1: return "WEAK" elif score == 2: return "MEDIUM" elif score == 3: return "STRONG" else: return "VERY STRONG"

🤖 Siri. Alexa. ChatGPT. All made of functions.
A function is a reusable block of code with a name. Every AI assistant is literally a function: takes your text → processes it → returns a reply. You're building one right now.
defreturnParametersArgumentsDefault Args*args **kwargsLambdaScopeGlobal/LocalRecursion
💡 Say this: "A function is a machine — you put something IN (argument), it does work, and gives something BACK (return value). ChatGPT's core logic is: take your message → decide what to say → return reply. That's literally a function. Let's build PyBot."
🔬 Anatomy of a Function — Every Part Explained
🏭
def keyword
Short for "define". Tells Python you're creating a named, reusable block of code.
📥
parameters
The inputs a function expects. Defined in the () when creating the function.
⚙️
body
The actual work — indented code block inside the def. Must be indented or Python throws an error.
📤
return
Sends a value back to where the function was called. Without return, gives None.
# Every part labelled: def greet(username, platform="Instagram"): # ↑name ↑param ↑default value msg = f"Welcome to {platform}, @{username}!" return msg # sends value back result = greet("riya_codes") print(result) # "Welcome to Instagram, @riya_codes!"
DRY Principle — Don't Repeat Yourself
Without functions, you'd copy-paste the same code 100 times. One bug = fix 100 places. With functions, write once, use everywhere. Fix once, fixed everywhere.

Instagram: The "send notification" logic is ONE function. Whether you get a like, comment, or follow — all three call the same send_notification() function with different arguments.
Scope — Local vs Global
Variables inside a function are local — only exist inside that function. Variables outside are global — visible everywhere.

📦 List vs Tuple vs Dictionary — When to Use Which
LIST [ ] — ordered, changeable
Use when:
You need an ordered collection to add/remove/loop through. Contacts, posts, comments, hashtags, search history.
viewers = ["Riya", "Arjun"] viewers.append("Dev") viewers.remove("Arjun") viewers.sort() viewers[0] # first viewers[-1] # last viewers[1:3] # slice len(viewers) # count
TUPLE ( ) — ordered, fixed
Use when:
Data won't change. GPS coords, RGB colour values, database primary keys. Faster than lists.
# Chennai — never changes location = (13.08, 80.27) lat = location[0] # 13.08 lng = location[1] # 80.27 # Story metadata — read only story = (12345, "2024-01-15", 24) # (post_id, date, hours_left)
DICT { } — key-value pairs
Use when:
You need named access to data. Look up by label (key) to get value. User profiles, app settings, JSON data.
profile = { "username": "riya", "followers": 45000, "verified": True } profile["username"] # "riya" profile.keys() profile.values() profile.items()
✨ List Comprehension — Python's Superpower
What it is: A compact, elegant way to build a list from other data in one line. Netflix uses this to filter movies. Instagram uses this to find trending posts. Every Python developer loves this.
# Normal way (4 lines): verified = [] for u in all_users: if u["verified"]: verified.append(u) # List comprehension (1 line!): verified = [u for u in all_users if u["verified"]]
# More examples: # Square all numbers: squares = [x**2 for x in range(10)] # Posts with >1000 likes: viral = [p for p in posts if p["likes"]>1000] # All usernames in uppercase: caps = [u.upper() for u in usernames]

UNIT V · 9 HRS · OBJECT ORIENTED PROGRAMMING
🏗️ Instagram is a class. You are an object.
Every Instagram account has the same structure but different data. That's a class — a blueprint. When you sign up, Instagram runs InstagramUser() and creates a new object just for you.
class__init__selfAttributesMethodsInheritancesuper()Method overridingData hiding __Encapsulation
💡 Say this: "Instagram has ONE blueprint called 'User'. When you sign up, Instagram creates a COPY of that blueprint with YOUR data. The blueprint is the class. Your account is the object (instance). Every user is a different object from the same class — that's OOP."
🏛️ The 4 Pillars of OOP — Instagram Edition
📦
Encapsulation
Bundle data (username, followers) and methods (post, follow) together in one class. Instagram's user data is hidden inside their server — only their methods can modify it.
🧬
Inheritance
Child class inherits from parent. BusinessUser inherits all features of InstagramUser and adds its own: analytics, ads, shopping tags.
🔄
Polymorphism
Same method name, different behaviour. post() on a personal account vs a business account both work but do slightly different things.
🔒
Abstraction
Hide complex internals. You tap "like" — you don't see the 47 database operations that happen. The complexity is abstracted away.

instagram_oop.py
class InstagramUser: def __init__(self, username, name): self.username = username self.name = name self.followers = 0 self.posts = [] self.total_likes = 0 self.__password = None # private def post(self, caption): self.posts.append({"caption":caption,"likes":0}) print(f"@{self.username} posted!") def follow(self, other): other.followers += 1 print(f"@{self.username} → @{other.username}") # Inheritance — BusinessUser extends InstagramUser class BusinessUser(InstagramUser): def __init__(self, username, name, brand): super().__init__(username, name) self.brand = brand def run_ad(self, budget): print(f"Running ad for {self.brand}!")
🔒 Data Hiding: Your Instagram password is stored internally as self.__password. The double underscore makes it private — you can't access it from outside the class. Only methods inside the class can touch it. That's data hiding / encapsulation protecting your security.
